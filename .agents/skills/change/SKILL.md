# プロジェクト変更ガイド

既存プロジェクトに対する変更（バグ修正・新機能追加・リファクタリング）を **7 つのフェーズ** に分けて管理します。
最新のフローでは、開発者の負担を減らすため、自律的な連続実行（ノンストップ実行）を基本としています。

## フェーズ一覧

| #   | フェーズ                        | プロンプト               | 成果物                                    |
| --- | ------------------------------- | ------------------------ | ----------------------------------------- |
| 1   | 提案 & 影響分析                 | `@skill change1` | `spec/PROPOSAL.md`                        |
| 2   | 仕様更新                        | `@skill change2` | 既存仕様ファイルの更新                    |
| 3   | レビュー & 整合性確認           | `@skill change3` | 各成果物の更新                            |
| 4   | タスク計画                      | `@skill change4` | `spec/TODO.md`, `spec/SESSION_CONTEXT.md` |
| 5   | 実装 (TDD)                      | `@skill change5` | コード + `spec/KNOWLEDGE_BASE.md`         |
| 6   | 変更レビュー & ドキュメント更新 | `@skill change6` | `README.md` 更新                          |
| 7   | アーカイブ & クリーンアップ     | `@skill change7` | `spec/archives/[YYYYMMDD_N]/`             |

## 特殊スキル（自動化・再修正・コミット）

| スキル | 説明 |
| --- | --- |
| `@skill change_auto` | Phase 2 から Phase 6 までをノンストップで連続実行します。一気に実装からレビューまで完了させたい場合に使用します。 |
| `@skill change_revise` | 各フェーズでの提示内容や、実装完了時の成果物に対するフィードバック（修正指示）を受け取り、再修正と再テストを行います。 |
| `@skill commit` | 変更完了後に Git コミット（Conventional Commits 形式）を自動生成し実行します。 |

## 標準的な進め方（一括実行フロー）

1. **提案**: `@skill change1` を呼び出し、変更内容を入力します。エージェントがヒアリングを行い `PROPOSAL.md` を作成して提示します。
2. **自動実行**: 提案内容に問題がなければ、`@skill change_auto` を実行します。エージェントが仕様更新から実装、最終レビューまでを一気に進めます。
3. **再修正**: 提示された成果物や実装コードに修正要望があれば、`@skill change_revise 〇〇を直して` と指示を出します。
4. **完了**: 全て問題がなくなったら、`@skill change7` でアーカイブを行い、`@skill commit` で作業を完了します。

※ 各フェーズ（`@skill change2` など）を個別に呼び出し、細かく確認しながら進める（個別実行フロー）ことも可能です。

## まずは Phase 1 から始めましょう

`@skill change1` を実行し、変更したい内容の概要を入力してください。
