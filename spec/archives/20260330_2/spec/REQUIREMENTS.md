# Product Requirements Document

## 1. Project Overview

- **Project Name**: ChatOps Framework
- **Vision**: Slack チャンネルへの投稿をトリガーに、プラグイン形式で拡張可能なコマンドを非同期実行し、結果を Slack に返す自前 ChatOps 基盤（DB をキュー兼ステート管理として一元利用）
- **Goals**:
  - 既存の Slack メッセージ受信 Hook・投稿プロキシ API と連携し、SaaS・Slack アプリ権限に依存しない完全自前の ChatOps 基盤を構築する
  - プラグインファイルを追加するだけでコマンドを増やせる拡張性を持つ
  - コマンド失敗時の自動リトライ機構を持ち、最終失敗は Slack へ通知する

## 2. Target Audience

- **主なユーザー**: インフラエンジニア・SRE などのオペレーター
  - Slack チャンネル上で `!コマンド名 [--オプション名 値...] [引数...]` の形式でコマンドを実行する（通常のシェルコマンドと同等の書式）
  - コマンドの実行結果を Slack スレッド上で受け取る
- **ユーザーの課題**:
  - サーバー監視・AI 解析などの操作を Slack から統一的に実行したい
  - コマンド追加のたびにフレームワーク本体を改修するコストを避けたい

## 3. Core Features (Scope)

### Must Have (MVP)

- **メッセージ受信 API**
  - 既存の Slack Hook 仕組みからメッセージ（チャンネル ID、スレッド情報、投稿者、本文、タイムスタンプ等）を受け取る HTTP エンドポイントを持つ

- **メッセージ永続化 + コマンド検出・ジョブ登録（同一トランザクション）**
  - 受信したメッセージをすべて DB に保存する
  - 保存と同時に本文を解析し、`!コマンド名 [--オプション名 値...] [引数...]` の形式のコマンドが含まれていれば**同一トランザクション内で DB にジョブレコードを登録**する（DB をキューとして利用）
  - ジョブレコードには「実行ステータス（pending / processing / done / failed）」「引数情報」「Slack スレッド会話情報」を保持する
  - これにより別プロセスによる DB スキャンが不要になり、コマンド検出が即時に行われる

- **非同期コマンド実行（ワーカー）**
  - ワーカーは **1 プロセス**で動作し、内部スレッドによって複数コマンドを同時並行実行できる
  - ワーカーが定期的に DB をポーリングし、`pending` 状態のジョブを複数取り出して各スレッドに割り当てる
  - 同時実行スレッド数の上限は設定値として変更可能とする
  - ポーリング間隔は設定値として変更可能とする（数秒〜十数秒の遅延は許容範囲）
  - 実行結果を既存の Slack 投稿プロキシ API 経由でスレッドに返信する

- **リトライ機構**
  - コマンド実行失敗時、DB のジョブレコードにリトライ要フラグ・失敗回数・失敗理由を記録し、ステータスを `pending` に戻す
  - 一定回数（設定値）までは自動リトライを行う
  - リトライ上限を超えた場合はステータスを `failed` に確定し、「最終失敗通知」を Slack スレッドへ投稿して処理を終了する

- **プラグイン形式のコマンド拡張**
  - 所定ディレクトリにプラグインファイルを追加するだけで新コマンドとして認識・実行できる
  - 各プラグインは「コマンド名」「説明」「実行ロジック」を定義するインターフェースに従う

- **`!help` コマンド**
  - 登録済みの全コマンド一覧とその説明を Slack に返すビルトインコマンドとして最初からサポートする

- **ダミーコマンドの同梱**
  - プラグイン実装のリファレンスとして、動作確認用のダミーコマンド（例: `!alert --host xxx web01`）を同梱する
  - ダミーコマンドは受け取った引数とスレッド情報をそのまま Slack に返す

### Should Have (v2)

- コマンド実行の監査ログ（誰が・いつ・何を実行したか）
- コマンド権限管理（特定ユーザー・チャンネルのみ実行可）
- 本番コマンドの実装（`!alert` による AI アラート解析など）

### Out of Scope (今回はやらない)

- Slack アプリ・Slack API の直接利用
- 外部 SaaS の利用（完全無料・自前で完結）
- 24 時間 365 日の高可用性・冗長化設計
- コマンド実行の監査ログ・権限管理
- AI 解析などの実コマンド実装

## 4. Non-functional Requirements

- **DBスキーマ管理**: マイグレーションツールによるリビジョン管理を行うこと。スキーマ変更はマイグレーションファイルとして記録し、バージョンアップ・ロールバックが可能な状態を維持すること
- **拡張性**: プラグインファイルの追加のみでコマンドを増やせること。フレームワーク本体の改修は不要
- **リトライ**: 最大リトライ回数は設定値として外部から変更可能であること
- **インターフェース整合性**:
  - 受信側: 既存 Slack Hook が送信する JSON フォーマットに準拠すること
  - 送信側: 既存 Slack 投稿プロキシ API の仕様に準拠すること
- **DB キュー**: 別途キュー専用ミドルウェアは使わず DB をキュー兼ステート管理として一元利用すること
- **ポーリング間隔**: ワーカーのポーリング間隔は設定値として外部から変更可能であること
- **非同期処理**: コマンドワーカーはメッセージ受信 API と独立して非同期で動作すること
- **応答遅延**: 数秒〜十数秒の遅延は許容範囲とする
- **ログ・トレーサビリティ**: すべてのログ出力に出力元ファイル名・行番号を含めること。メッセージ受信からジョブ実行までの処理フローを単一の trace_id で横断的に追跡できること。trace_id はジョブ登録時に DB に保存し、ワーカーが復元することで複数プロセス間を跨いだトレースを実現すること
